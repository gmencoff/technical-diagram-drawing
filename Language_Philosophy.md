# Language Philosophy

## Mission

The primary goal of this project is to make it easy for **AI systems to
generate accurate, meaningful technical diagrams**.

Large language models are becoming increasingly capable of reasoning
about engineering systems, but today they have no semantic language for
communicating those ideas visually. Existing graphics formats focus on
drawing commands rather than engineering meaning, making them difficult
for AI to author reliably and difficult for humans to maintain.

This project aims to define a semantic language that allows humans and
AI to describe engineering systems using domain concepts rather than
graphical primitives. The renderer is then responsible for transforming
that semantic description into one or more visual representations.

**Graphics are an output---not the language itself.**

------------------------------------------------------------------------

# Semantic Rather Than Graphical

The language should describe **what exists**, not **how it is drawn**.

Rather than specifying coordinates, paths, and graphical primitives,
users describe engineering concepts.

``` yaml
objects:
  - type: antenna.ULA
    id: rxArray
    numElements: 4
```

The renderer determines how that object is visualized. This philosophy
should apply throughout the language.

------------------------------------------------------------------------

# Domain Objects

Objects should represent meaningful engineering concepts rather than
graphical primitives.

Examples include:

-   `antenna.Element`
-   `antenna.ULA`
-   `rf.LNA`
-   `rf.PhaseShifter`
-   `radar.Transmitter`
-   `radar.Target`

The language should expose the terminology engineers naturally use.

------------------------------------------------------------------------

# Composability

Simple semantic objects should compose into more complex semantic
objects.

For example, `antenna.Element` is the primitive building block, while
`antenna.ULA` is constructed by expanding into multiple
`antenna.Element` objects.

This promotes reuse while allowing higher-level abstractions to remain
concise.

------------------------------------------------------------------------

# Authoring Model as the Source of Truth

The authoring document is the canonical representation of a diagram.

Everything else---including generated objects, layout, and rendered
graphics---is derived from this representation.

This keeps diagrams concise, understandable, easy to generate with AI,
and suitable for version control and long-term maintenance.

------------------------------------------------------------------------

# Properties Versus Features

A clear distinction exists between information supplied by the author
and information generated by the renderer.

## Properties

Properties configure an object.

Examples:

-   `numElements`
-   `radius`
-   `frequency`

Properties appear in the authoring language.

## Features

Features are intrinsic capabilities exposed by an object.

Examples:

-   `center`
-   `port`
-   `axis`
-   `element[3]`

Features are generated automatically and are primarily used for
references, annotations, layout, and rendering.

------------------------------------------------------------------------

# Intelligent Defaults

The language should minimize the amount of information required to
produce useful diagrams.

Reasonable defaults should always be preferred over requiring additional
syntax.

Simple objects should immediately produce meaningful figures without
requiring layout or styling information.

------------------------------------------------------------------------

# Progressive Complexity

The language should scale naturally from simple educational figures to
large engineering systems.

New users should only need a handful of concepts, while advanced users
should be able to build sophisticated diagrams without changing the
underlying philosophy.

------------------------------------------------------------------------

# LLM-First Design

The language should be intentionally designed for AI generation.

This means:

-   Predictable schemas
-   Deterministic defaults
-   Consistent naming
-   Minimal verbosity
-   Minimal ambiguity
-   Very few special cases

A valid specification should be easy for an LLM to produce correctly
without iterative refinement.

------------------------------------------------------------------------

# Round-Trip Editing

The language should support both textual authoring and graphical
editing.

Whenever possible, graphical edits should modify semantic properties
rather than graphical geometry.

For example, increasing the spacing of an antenna array should update
the array's spacing property rather than moving individual antenna
elements independently.

This preserves semantic meaning and allows diagrams to be shared,
regenerated, and edited by both humans and AI.

------------------------------------------------------------------------

# Separation of Meaning and Presentation

The semantic meaning of an object should be independent of its visual
representation.

For example,

``` yaml
objects:
  - type: antenna.Element
    id: tx
```

describes the existence of an antenna element, but not how that antenna
should appear.

Different renderers may produce:

-   Publication-quality figures
-   Presentation graphics
-   Educational illustrations
-   Schematic views
-   Interactive editors
-   Accessibility-focused visualizations

All are generated from the same semantic description.

------------------------------------------------------------------------

# Extensibility

The language is intended to evolve by expanding semantic knowledge
rather than graphical capabilities.

New functionality should be introduced by defining richer engineering
objects or extending existing ones with additional semantic meaning.

For example, future versions of `antenna.Element` may support
polarization, radiation patterns, impedance, or mutual coupling without
changing the overall philosophy.

Entirely new technical domains can be introduced through namespaces such
as:

-   `antenna.*`
-   `rf.*`
-   `radar.*`
-   `signal.*`
-   `optics.*`
-   `power.*`
-   `control.*`

The language should therefore evolve by **capturing more engineering
knowledge**, not by accumulating additional drawing commands.

------------------------------------------------------------------------

# Separation of Responsibilities

The system should maintain clear architectural boundaries.

-   The authoring language describes engineering intent.
-   Semantic expansion generates intrinsic features.
-   The layout engine computes geometry.
-   The renderer produces graphical primitives.
-   The output format (SVG, PowerPoint, Canvas, etc.) is simply one
    possible representation of the semantic model.

Each stage should remain independent, making the system easier to
understand, extend, test, and maintain.
